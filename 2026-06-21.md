### 1. Cross-browser Service Order divergence (76 vs 69 vs 75) — ROOT CAUSE + FIX
Same user, different browsers saw different job lists/totals. ROOT: job deletions were never durable.
- `handleDeleteJob` (App.tsx ~1757) wrote `solarflow_deleted_job_ids` to **localStorage only** and never deleted the server `job:{id}` row.
- Unlike `deleted_customer_ids` (synced), `deleted_job_ids` was **never pushed or pulled**. So every device re-pulled deleted jobs, and crucially a stale browser **re-pushed** them: a deleted row has no server copy, so `dropStaleRows` waves it through as "new" → resurrection. Logs proved it: I deleted 18 ghost rows (86→68) and within the hour the server climbed back to 75 with 7 tombstoned ids LIVE again + a 14:18:14 batch re-push.
- Each browser = independent localStorage accumulation (incremental pull + mergeRemote keeps local-only jobs), holding a different ghost subset → divergent totals.

FIX (syncEngine.ts, commit f461d14, mirrors the customer tombstone path):
1. pull + union `deleted_job_ids` into local (KV `.in()` list + handler).
2. push `deleted_job_ids` as a meta row in `pushToSupabase`.
3. **exclude tombstoned ids from the job push** (`deletedJobIdSet`) so a stale browser can't recreate them — the key new guard.
4. bump `SYNC_CURSOR_VERSION` 2→3 → one full reconcile pull per device on next load.
Verified: `tsc -b` clean, 169/169 tests (job-tombstone filter already covered at syncEngine.test.ts:190).

Server reconcile (one-time, via REST as cesar.jurado): seeded `deleted_job_ids` with the 18 ids from change_log `op_type=job.delete`, deleted the 18 ghost `job:{id}` rows.

RULE: every per-record synced entity MUST sync its tombstone (push + pull + union) AND exclude tombstoned ids from push, or deletions diverge + resurrect. See [[sync-job-tombstones]].
