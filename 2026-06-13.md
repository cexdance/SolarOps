## e9f0c3c - CRITICAL sync bug: client-clock pull cursor skipped records forever (Jaime/Alfonso R2)

After d5607e1 (auto-dispatch + data fix), Jaime STILL couldn't see Alfonso despite many
refreshes. Verified server-side via direct Supabase query (await import('/src/lib/supabase.ts')
in preview eval): the app_data row job:job-1780948038009 WAS correct (contractor-4, scheduled,
fresh updated_at). So the write persisted - the failure was the READ/sync path to Jaime.

ROOT CAUSE: incremental pull cursor. pullFromSupabase did `since = getLastSync()` then
`pullPrefix(..., since)` -> `q.gt('updated_at', since)`. But `setLastSync(now)` used the CLIENT
clock (`new Date()`), while app_data.updated_at is SERVER time. A device whose clock runs ahead
of the DB stores a FUTURE `since`, so every record the server writes in that skew window is
`<= since` => permanently skipped on every refresh. Compounded: pushToSupabase ALSO did
setLastSync(now), and the syncCycle drains the outbox (push) right BEFORE the pull - so a
contractor's own push shoved the read cursor into the future and the next pull skipped all other
devices' recent writes. This is the textbook "admin sees it, contractor refreshes forever and
never gets it" bug.

FIXES (syncEngine.ts):
- Pull cursor now derived from the MAX server `updated_at` of the rows actually returned (pullPrefix
  stamps value.updatedAt = app_data.updated_at), minus a 5s safety overlap. If nothing new returned,
  `since` is left untouched (never jumps to client-now). Immune to client clock skew.
- pushToSupabase no longer calls setLastSync - a push must not move the read cursor (re-pulling our
  own writes once is a harmless idempotent merge; the pull owns the cursor).
- SYNC_CURSOR_VERSION ('2') + ensureCursorVersion(): first pull of this build clears the stale
  cursor ONCE and does a full resync, recovering rows a future cursor had been skipping (devices
  already stuck need this; the forward fix alone wouldn't rewind their cursor). VERIFIED in preview:
  set a +1h future cursor + old version, reload -> cursor reset to a server-derived value (behind
  client-now), version='2', Alfonso came through correct.

PATTERN: NEVER set an incremental sync cursor from the client clock - use the server's own
timestamps (max updated_at returned), and a push must not advance the read cursor. Bump a cursor
version to force a one-time full resync when changing cursor semantics. tsc/build clean, 166 tests.
Jaime gets Alfonso once his phone loads this build (stale cursor resets -> full pull).
