## Deploy 13bee49 - Fix React.Fragment data-matrix-id console flood

**Commit:** `13bee49` (pushed to `main`). Follows 4372060 / 5627463 earlier today.

**Files modified:**
- `solarflow-dashboard/src/components/Customers.tsx`
- `solarflow-dashboard/src/components/ui/ActivityFeed.tsx`

**Problem:** The preview/build instrumentation injects a `data-matrix-id` prop onto JSX elements. React rejects it on a `React.Fragment` ("Invalid prop supplied to React.Fragment - can only have key and children"), flooding the console 500+ times whenever the customer detail panel rendered. The grep for `data-matrix-id` in src returns nothing because it is injected by tooling, not written in source.

**Root cause:** Two keyed Fragments rendered inside `.map()` loops:
1. `Customers.tsx` WO pipeline (`WO_PIPELINE.map`, was ~L4167): `<React.Fragment key={stage.key}>`.
2. `ActivityFeed.tsx` `MentionBody` plain-text segment (L144): `<React.Fragment key={i}>{part}</React.Fragment>`.

**Fix (minimal, layout-neutral):**
- Pipeline Fragment -> `<div key={stage.key} className="contents">`. Tailwind `contents` = `display:contents`, so the div does not affect the flex row (the `flex-1` connector + node still lay out identically) but gives the instrumentation a real DOM node to tag.
- MentionBody Fragment -> `<span key={i}>{part}</span>`. Inline, so text flow inside the `<p>` is unchanged.

**Verification gotcha worth remembering:** The MCP preview console buffer (CDP-captured) accumulates across reloads and is NOT cleared by an in-page `console.error`/`console.clear()`. So `preview_console_logs` kept showing stale warnings even after the fix loaded. Two reliable signals instead:
1. Fetch the served module from vite in-page (`fetch('/src/.../File.tsx?import')`) and grep the transformed source for old-vs-new code.
2. Monkeypatch `console.error` in-page to a counter, reset it, force re-renders, then read the counter. Counter was 0 -> no new Fragment warnings.
Also: HMR sometimes fails to apply edits to large files ("[vite] Failed to reload ...") - a full `location.reload()` is needed, and even then read the served module to confirm, do not trust the buffered console.

**Pattern:** When tooling spreads attributes onto Fragments and you cannot change the tooling, replace the Fragment with `div.contents` (block context) or `span` (inline context) - both accept the prop with zero layout change.

**Visual impact:** none (console-only). Skipped `/snap-ui` per the zero-visual-impact exception.
