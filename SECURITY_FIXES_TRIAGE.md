# Security Fix Spec - Triage for SolarOps (Vite SPA + Vercel + Supabase)

The spec (`solarflow_fix_spec.md`) was written assuming Next.js + cookie sessions.
SolarOps is a **Vite SPA** with **Bearer-JWT auth** (token in localStorage) and
**Vercel serverless functions** in `solarflow-dashboard/api/`. This maps each task
to what is real for THIS stack.

Branch: `security/spec-fixes`

## Status by task

### T1 - Open self-service registration (CRITICAL)
- **1C/1D (code): ALREADY SATISFIED.** No `supabase.auth.signUp` exists anywhere in `src/`.
  The contractor "Apply to join" flow (`ContractorRegister.tsx`) does not create auth accounts.
  The only signup vector is the Supabase endpoint being open at the platform level.
- **1A (YOU - Supabase Dashboard, do this first):** Authentication > Providers > Email >
  toggle OFF "Allow new users to sign up". This is the real fix.
- **1B (YOU - optional defense-in-depth):** the domain-allowlist SQL in the spec works, but a
  trigger on `auth.users` can break auth flows; prefer Supabase **Auth Hooks** (Before User Created)
  if available. 1A alone closes the hole.

### T2 - Brute-force rate limiting (HIGH)
- **2B is a NO-OP for this stack - do NOT add it.** Login calls `supabase.auth.signInWithPassword`
  **directly to `cjmhfagkkayelcsprbai.supabase.co`** (see `App.tsx:173`, `LoginScreen.tsx:82`,
  `ContractorLoginScreen.tsx:62`). It never touches our Vercel `/api`, so a Vercel middleware /
  `vercel.json` limiter would never see a login. (Also `vercel.json` `routes.rateLimit` is not a
  supported feature.)
- **2A (YOU - Supabase Dashboard, the real fix):** Authentication > Rate Limits > cap sign-in
  attempts; enable CAPTCHA (Turnstile). This is the only effective control for the real login path.

### T3 - version.json exposure (MEDIUM) - DECISION NEEDED
- **Do NOT delete it.** `version.json` is generated by `vite.config.ts` and consumed by BOTH
  `index.html` (pre-React stale-cache guard) and `useVersionPoll.ts` (in-app update prompt).
  Deleting it breaks auto-update / cache-busting.
- Real concern is only the `sha` field. Options:
  - (a) Leave as-is if the GitHub repo is **private** (SHA exposure ~ nil).
  - (b) Strip `sha` and replace the git-sha portion of `buildId` with a random hash (keeps the
    version feature, removes the source-pinpoint). I can do this if you want - it touches the
    version system, so I left it for your call.

### T4 - API routes 500 instead of 401 (MEDIUM) - FIXED (code)
- Root cause in `api/_auth.ts`: the `!SERVICE_ROLE_KEY` -> 500 check ran BEFORE the `!token` -> 401
  check, so anonymous probes got 500. **Reordered:** no token -> 401 always; 500 only when a token
  is present but the server key is missing. `/api/health` already needs no auth.
- NOTE: production currently 500s on ALL `/api/*` because the latest deploy has not shipped (env
  guard blocked it / `SUPABASE_SERVICE_ROLE_KEY` not yet correct). Once that key is set and the
  deploy succeeds, health -> 200, users/solaredge -> 401 unauth. This fix is necessary but the
  deploy/env step (already pending from earlier) is what makes the acceptance criteria pass live.

### T5 - Server-side route guard (LOW) - NOT APPLICABLE as written
- The spec's cookie-based edge middleware cannot work: this SPA stores the Supabase session in
  **localStorage, not cookies**, so there is no cookie for an edge function to read. `/dashboard`
  etc. are the static `index.html` shell (no sensitive data); the real security boundary is the
  API (`requireUser`) + RLS. Implementing true server guards means migrating to cookie sessions
  (large change). Recommend: rely on T4 (API auth) + T6 (RLS). Defer the SSR guard.
