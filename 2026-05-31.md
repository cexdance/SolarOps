## What Broke
Contractor Carlos Valbuena worked on US-15200 (Clarance Harris) and US-15320 (Ana Henriquez) and all photos were lost. Investigation confirmed zero `cj-*` files in Supabase Storage for any contractor job, ever. Three compounding bugs caused this.

## Bugs Fixed

### CB-1 - Start Work button hidden behind device bottom bar
`src/index.css` + `src/components/contractor/JobDetail.tsx`
- `.pb-safe` and `.safe-area-pb` utility classes were referenced but never defined
- Added them using `env(safe-area-inset-bottom)` in `@layer utilities`
- Added `z-40` to the fixed CTA bar

### CB-2 - Failed uploads permanently destroyed photos
`src/components/contractor/JobDetail.tsx`
- A connectivity failure left photos as base64 `data:` URLs in state
- `sanitizeContractorJobsPayload` stripped them before the sync push
- Fix: `retryPendingPhotoUploads()` sweeps for base64 URLs on mount and `online` event, re-uploads, swaps in Storage URL

### CB-3 - Whole-blob LWW clobbered other devices' jobs and photos
`src/lib/syncEngine.ts`, `src/App.tsx`, `src/types/contractor.ts`
- `pushKeyValue` did a blind upsert of the entire contractor_jobs array
- `pullFromSupabase` did a blind `localStorage.setItem`, wiping local-only jobs and photos
- Fix: `mergeContractorJobs(local, remote)` - per-record LWW by `updatedAt ?? assignedAt`, photo union per category, one-sided jobs preserved
- Merge-on-write: fetch remote, merge, sanitize, upsert
- Merge-on-read: merge remote into local instead of overwriting
- `handleContractorJobUpdate` now stamps `updatedAt` on every edit

### CB-4 - Duplicate contractor jobs for one admin work order
Closed as a side effect of CB-3. `mergeContractorJobs` deduplicates by `sourceJobId`, keeping newest and unioning photos.

## Data Loss Note
Carlos's US-15200 and US-15320 photos are unrecoverable from the cloud. Only his device's localStorage (if untouched) may still hold them. With CB-2 deployed, opening the app on his device will auto-retry the uploads.

## Photo Durability Chain (post-fix)
1. Take photo -> base64 preview shown immediately
2. Upload attempt -> on success, swap to Storage URL
3. On failure, base64 stays in state
4. CB-3 merge preserves local dataUrl across pulls
5. On reconnect, retry sweep uploads it, swaps in Storage URL
6. Storage URL propagates to all devices via per-record merge

## Patterns to Reuse

- `mergeContractorJobs` pattern: union arrays per key, LWW by timestamp for scalar fields
- Photo retry: `useCallback` + `retryInFlight` ref + `window.addEventListener('online', ...)`

### Why every prior fix "kept propagating" (never reached users)
Production was frozen at the May 29 build (db 05-29, v1.7.4). The deployed
app kept showing the old kanban ribbon because **the production build was

### Root cause: `tsc -b && vite build` failed
- `tsc --noEmit` (root) does NOT replicate the build. Only `tsc -b`
  typechecks the referenced project, which includes `src/__tests__`.
- `src/lib/customerStore.ts` used `CustomerStatus`, `InteractionType`,
  `InteractionOutcome` but never imported them (TS2304/TS2552). PROD code.
- `tsconfig.app.json` included `src/**` (incl. tests); test-only cast
  errors (`as WOPhoto` / `as Job`) failed the prod build.

### Fix
- Added the missing imports to customerStore.ts.
- `tsconfig.app.json`: excluded `src/__tests__` + `*.test.ts(x)` from the
  production build (tests run under vitest, not shipped). Durable: test
  type errors can never block a deploy again.
- Hardened the test casts to `as unknown as`.
- Verified: `pnpm run build` succeeds, stamps v1.7.5 + db 05-31, old kanban
  strings absent from emitted bundle, 109/109 vitest pass.

## ROOT CAUSE of "Storage full" — HEIC stored raw (b566bc8)
- iPhone photos are HEIC. compressImageToDataUrl/Blob preemptively bailed on HEIC
  and stored the raw 3-8MB file as base64 → a single photo could blow the ~5MB
  localStorage cap. Removed the bail; loadBitmap decodes HEIC (createImageBitmap/
  <img>) with raw fallback only on real failure. New photos now ~150-300KB.
- Applies to NEW photos. Existing bloat clears when the device gets online
  (uploads swap base64→https) or via Export backup + fresh sign-in.
