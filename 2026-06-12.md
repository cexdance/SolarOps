## DEPLOYED 4179448 - CRITICAL sync fix: unhydrated session mass-overwrite + data recovery

**INCIDENT.** User reported: comment on US-15339 yesterday showed in the mention bell but vanished from the customer. Trace: the note SAVED correctly (change_log 01:28:37 payload has it in `customer.activityHistory`; user then renamed Iki Dugno -> Meagen Murphy, new email+phone, all logged through 01:31:20). At **14:26:35Z a mass sweep rewrote 269 customer rows, 225 as seed-shaped skeletons** (no activityHistory/files, empty emails), reverting the note, the rename, and 360 of the 06-11 audit field fixes. The mention survived because fireMentionNotifications writes to the notifications table independently of the customer save.

**ROOT CAUSE.** `pushToSupabase`'s dirty map (`_lastPushed`) is in-memory and starts EMPTY each session; `saveData` (debounced 500ms on ANY data change, including mount) -> `dbSet` -> `pushToSupabase` therefore mass-pushes the ENTIRE local state the first time it runs, and `pushRows` re-stamped `value.updatedAt = now`, making stale/seed data win every later LWW merge on every device. A session holding seed/stale localStorage (fresh browser, cleared storage) + an auth session = full DB wipe. Compounding bugs: useSyncEngine never pulled on mount (only 5min/focus/online); App's startup merge only adopted remote when record COUNTS changed (length-only compare), so same-count content updates left React state stale and the next edit pushed stale state back out.

**FIX (4179448, all in syncEngine.ts / useSyncEngine.ts / App.tsx):**
1. **Session push gate**: no push until one successful `pullFromSupabase` this session (`markSessionPulled`); blocked pushes mark pending, outbox retries.
2. **Stale-write guard** (`dropStaleRows`): before upsert, fetch server `value->>updatedAt` for dirty keys; DROP rows whose server copy is strictly newer; dropped rows markClean (re-arm on real edit). New rows + equal stamps pass. Fails closed.
3. **Edit-time `updatedAt` stamp** in handleUpdateCustomer (edits always beat the guard).
4. Startup merge ALWAYS adopts merged state; useSyncEngine pulls on mount.
7 new tests in `syncPushGate.test.ts` (gate, guard newer/older/equal/new, re-arm). 158 green.

**RECOVERY (scripts/04-sweep-restore.mjs + inline reapply, backups in address-audit/):**
- 32 customers restored from latest pre-sweep change_log full payloads (notes/files/identity), incl. US-15339 (Meagen Murphy, 13 activities, @cruzfernandez note, phone 678-974-9538) and Jason Brunson (21 activities, 5 files). Logged as `customer.sweep_restore`.
- 182 customers re-patched with the user-approved 06-11 audit fixes the sweep reverted (136 addresses, 103 phones, 116 emails, 5 SE links). Logged as `customer.audit_reapply`.
- Remaining ~190 sweep-touched rows had no pre-sweep change_log payload; their enrichment (if any) predates logging and is unrecoverable from server data. Backup of pre-restore state: `address-audit/sweep-restore-backup.json`.

**PATTERNS:** mention bell != data saved (notifications table is a separate write); change_log full payloads are a per-record point-in-time backup, use as restore source; any new push path MUST respect the session gate + stale guard; `value.updatedAt` is EDIT time and the LWW key, never blanket-restamp stale records.
