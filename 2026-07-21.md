### 1. Inventory never synced, fixed with a union merge (`7de93ca`)

Cruz added items to Inventory from his phone; they never showed up anywhere else. Root cause: `saveInventory()` called `dbSet('solarops_inventory', items)`, but `db.ts` only routes `solarflow_data` and keys in `KV_SYNC_KEYS`. `solarops_inventory` was in neither, so the call was a silent no-op. The read side was equally dead: `dbGet` is a deprecated stub hardcoded to return `null`, and `syncInventoryFromDB()` had zero callers anywhere in the codebase. The file's own header comment falsely claimed "localStorage + Neon sync."

**Rescue attempt:** checked live Supabase directly. Zero `app_data` rows matching `%inv%`, zero `change_log` payloads with inventory fields, none of the 27 logged `op_type`s touch inventory. Nothing recoverable server-side, the items exist only in whatever device added them (Cruz's phone had 652 change_log rows and was active same-day, so his copy is presumed intact pending his next login).

**Why the obvious fix (just add the key to `KV_SYNC_KEYS`) was dangerous on its own:** unrecognized KV keys fall through to a blind remote-overwrite branch on pull. The sync engine has a hard push gate (pull-before-push, from the 2026-06-12 incident), so the very next time any device with local-only inventory items opened the app, its pull would have overwritten localStorage with whatever the DB had (nothing, or another device's seed), silently destroying the local-only items before they ever got pushed.

**Fix, shipped as one unit:**
- `KV_SYNC_KEYS` gains `'solarops_inventory'`
- `mergeInventoryItems()` (new, `syncEngine.ts`): union by id, newer `updatedAt`/`createdAt` wins, one-sided items are ALWAYS kept, modeled on the existing `mergeAddressCleanupItems` pattern
- Wired into **both** write paths: the pull branch AND the Realtime broadcast handler. The Realtime handler was found to blind-overwrite every KV key with no merge at all, a second copy of the same bug independent of inventory.
- `InventoryItem.updatedAt` added (optional, falls back to `createdAt` for legacy rows with none)
- `saveInventory()` stamps `updatedAt` only on items whose content actually changed vs. the previous save, a blanket stamp on every save would make whichever device saved most recently win every field, which is the exact clobber the merge exists to prevent
- Deleted the dead `dbGet`/`syncInventoryFromDB` path and the false "Neon sync" header comment
- Deletions are NOT propagated: inventory has no tombstone list, and with local-only history on every device a missing item currently means "this device never had it," not "deleted." Real deletes need the same tombstone treatment jobs got ([[sync_job_tombstones]]).

New test file `__tests__/inventoryMerge.test.ts`, 8 cases, including the actual regression scenario: local-only items on one side must survive a merge where the other side has never seen them.

### 2. Notifications search input: invisible text (`ba6e5cb`)

User-reported via screenshot: the notifications panel search box showed no visible text. Root cause: the panel dropdown lives inside `<header className="bg-slate-900 text-white">`. The dropdown resets its own background to white but never resets `color`, so the header's white text cascaded straight into the search `<input>`, which had no explicit text-color class, white text on a near-white `bg-slate-50` fill. Every other input in `Layout.tsx` sets color explicitly; this one didn't. One-line fix: added `text-slate-800 placeholder-slate-400`.

### Patterns worth reusing

- Before declaring any "sync" bug root-caused, check both the write path (`dbSet`) AND the read path (`dbGet`/whatever pull function) AND whether that pull function has any callers at all. This bug had all three broken simultaneously and a comment claiming otherwise.
- A KV key with more than one writer (phone + desktop + tech laptop) MUST have a union-merge branch before joining `KV_SYNC_KEYS`. The default branch is remote-wins-blind, and the push gate means the danger window is literally "the next time anyone opens the app."
- The Realtime broadcast handler and the pull handler are two SEPARATE code paths that must each apply the same per-key merge logic. Confirmed today that `solarops_inventory` needed the fix in both places; a background task (`task_94f91d83`, still running as of this note) was filed to check whether `crm_data`, `contractor_jobs`, and `address_cleanup` have the same gap in the Realtime path specifically.

## Deploy 71de75f - inventory per-location stock, synced tools, office-confirmed field pulls

**Live:** v1.7.5.5+71de75f, builtAt 2026-07-21T05:39:03.594Z, verified on solarflow-dashboard-sooty.vercel.app.

### Why
Follow-on from the ba6e5cb inventory-sync fix earlier today. A review of RMA / Inventory / Customers data flow found the earlier fix was only half the story: the receive side merged, the send side did not.

### Data-integrity fixes (the important part)
1. **`pushKeyValue` was a blind overwrite for every key except contractor jobs.** `solarops_inventory` and `solarops_address_cleanup` upserted the local blob straight over remote, so any device with a stale tab wiped whatever another device wrote since. Now merges against remote before upsert.
2. **`InventoryModule` never re-hydrated.** `useState(() => loadInventory())` at mount, and `solarops_inventory` was absent from both listeners in `useSyncEngine.ts`. A warehouse tab open all day operated on a mount-time snapshot and then pushed it (feeding bug 1). Now listens for `solarflow-remote-update`; the Realtime KV branch in `syncEngine.ts` now emits that event too, since `onKV` only covers keys `useSyncEngine` holds in App state.
3. **Inventory deletes resurrected.** Delete was `items.filter(...)`, and the union merge cannot tell a shorter array from "this device never had it". Deletes are now `deletedAt` tombstones riding the normal newest-wins merge. `saveInventory` re-attaches tombstones on every save, otherwise an ordinary save drops them.

### Pattern worth reusing: KV_MERGERS
The standing rule ("a multi-writer KV key needs a union merge in BOTH pull and Realtime handlers BEFORE joining KV_SYNC_KEYS") was enforced by comments, and it **had already drifted** - that drift is exactly bug 1. Replaced three near-identical branches with a single `KV_MERGERS` registry consulted by pull, Realtime **and** push. `src/__tests__/kvMergers.test.ts` fails the build if a multi-writer key has no merger. Adding a new multi-writer key is now one line and is merged everywhere by construction.

### Features
- `InventoryItem.stockByLocation: Record<string, number>` - units per location. `quantity` stays the **maintained total**, deliberately, so every existing valuation / low-stock / catalog read was untouched. Legacy rows migrate on load via `ensureStock`, idempotent. A location at 0 is dropped from the map because location keys ARE the location list.
- Locations **derive from the data** (`knownLocations`), not from a synced settings key. Avoids a new KV key and therefore a new merge branch. Contractor locations use `Contractor: <name>`, matching what the receiving modal already writes, so pre-existing contractor stock lands in the same bucket.
- Contractor picks a part from stock on the WO report -> records `inventoryItemId` + `fromLocation` only. Office clicks **Apply to stock** in the Inventory equipment tab. `applyPartsToInventory` is pure and idempotent (`appliedToInventoryAt` guards double-apply). Contractors are never writers on the inventory blob.
- **Tools were `useState(demoTools)` with no store at all** - every edit, status change and assignment lost on reload, invisible to every other device. New `src/lib/toolStore.ts` with the same union merge + tombstones, plus contractor check-out/check-in.
- Contractor RMA case entry (SolarEdge / other vendors) on the WO report. Mirrors into the admin job via `mergeRmaEntries`, a UNION not an assignment: each side only knows its own half, so assigning would delete the other side's entries.
