NOTE: this is the realization of the earlier flagged follow-up (move provenance images off inline dataURL to Storage). The existing item `imageUrl` (Add/Edit ImageUploader) is still inline dataURL — same quota risk if many item thumbnails added; consider migrating those too.

## Security spec triage + T4 auth-order fix (DEPLOYED e0f1e72)

### Commit pushed to main
- `e0f1e72` fix(auth): return 401 before 500 for unauthenticated API requests (T4) - range `4c68c1f..e0f1e72`.
- History verified linear/intact: `f7e1d83` -> `4c68c1f` (favicon) -> `e0f1e72`. No reverts.

### Context
User provided `~/solarflow_fix_spec.md` (6 pentest tasks). The spec was written assuming Next.js + cookie sessions; SolarOps is a Vite SPA with Bearer-JWT (localStorage) auth + Vercel serverless functions. Most tasks required translation, not literal application.

### Triage outcome (full detail in repo `SECURITY_FIXES_TRIAGE.md`)
- **T1 (open signup):** No `supabase.auth.signUp` exists in `src/` - already satisfied in code. Real fix is the Supabase dashboard "Allow new users to sign up" toggle (user action). 1B domain hook: user declined.
- **T2 (rate limit):** Login calls `signInWithPassword` DIRECTLY to Supabase (App.tsx:173, LoginScreen.tsx:82, ContractorLoginScreen.tsx:62), never hits our `/api`. Spec's Vercel middleware = no-op. Real fix = Supabase Rate Limits + CAPTCHA (dashboard).
- **T3 (version.json):** Do NOT delete - it powers the stale-cache/auto-update guard (vite.config.ts -> public/version.json, consumed by index.html + useVersionPoll.ts). Only `sha` is exposure. User chose "decide later".
- **T4 (500 vs 401):** FIXED. `api/_auth.ts` checked `!SERVICE_ROLE_KEY` (500) before `!token` (401), so anon probes got 500 + config leak. Reordered: no token -> 401 always; 500 only when a token is present but key missing. Updated apiResilience.test.ts (split into 401-unauth and 500-with-token cases). 123/123 pass.
- **T5 (SSR route guard):** Not applicable - SPA uses localStorage not cookies, so cookie edge-middleware can't work. Boundary is API auth (T4) + RLS (T6). Deferred.
- **T6 (RLS audit):** User action (Supabase SQL).

### Pattern worth reusing
- Shared auth guards should reject unauthenticated callers (401) BEFORE any server-config check (500). Never return a config-state-leaking 500 to an anonymous request.
- Always ground an externally-authored security spec against the actual stack (auth mechanism, deploy target) before applying - half of this spec was Next.js/cookie-specific and would have added dead/broken code.
