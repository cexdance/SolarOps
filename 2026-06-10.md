## SolarEdge sync 500 fixed (root /api build chain) + contractor multi-view + Dispatch Map

### SolarEdge 500 root cause + fix (commits `0489ac7`, `73d23c6`, `e865d8d`, `a9918b7`)
The live SolarEdge alert sync returned 500 (`FUNCTION_INVOCATION_FAILED`). Production deploys the **repo-root `/api`** tree. Commit `d0adbaf` (web-push) created a root `package.json` listing only `docx` + `web-push`, which cascaded:

1. **`@vercel/node` missing** , Vercel's function builder now treats root as an npm project and typechecks `/api/*.ts`; every function imports `@vercel/node` but it was undeclared -> all 13 functions failed `TS2307` -> every deploy Errored -> production stuck on a stale broken build serving the 500. Fix: added `@vercel/node` + `@types/node` to root `package.json` (`73d23c6`).
2. **Two masked type errors surfaced** once typecheck ran: `notify.ts` cast `Record<string,unknown>` -> web-push `PushSubscription` (fixed via `as unknown as`), `ups-tracking.ts` used `VercelResponse<T>` (not generic in @vercel/node v5; dropped the type arg). (`e865d8d`)
3. **Hobby 12-function cap** , root `/api` was 13 functions (push-subscribe was the 13th); build completed but "Deploying outputs" Errored. Consolidated the 3 unused Xero endpoints (zero frontend callers) into one action-routed `api/xero.ts` -> 11 functions. (`a9918b7`)
4. Earlier `0489ac7` added `.js` extensions to the **dash** `/api` tree's ESM relative imports (`./_auth` -> `./_auth.js`) , ESM hygiene; dash tree is not the canonical deploy but fixed regardless.

**Verified live:** `/api/solaredge?path=/sites/list` -> 200 (308 sites). `/api/users` -> 401 (was 500), `/api/push-subscribe` -> 405 (exists), `/api/trello-card` -> 400. Production alias = `solarflow-dashboard-sooty.vercel.app`.

PATTERN: a repo-root `package.json` flips Vercel's function builder into typecheck mode; it must declare `@vercel/node`. Build-completes-then-"Deploying outputs"-Error with no build error = the Hobby 12-serverless-function cap. Root `/api` is the canonical tree (native fetch, no `_auth`); dash `/api` is not deployed.

## Customer address cascade to contractor WO + SO pay-section redesign (commits `ee1a265`, `d2ed3d6`)

### Address edit now reaches the dispatched contractor WO (`ee1a265`)
Bug: editing a customer's address (US-15313) did not update the contractor's Work Order. The contractor view is a LIVE projection `toContractorJobView(job)` which reads `job.siteAddress` (city/state/zip hardcoded ''), and the dispatched `ContractorJob` is a snapshot taken at dispatch; but `handleUpdateCustomer` only rewrote the customer record, so neither updated.
Fix in `App.handleUpdateCustomer`: detect an address change (vs `data.customers` prev), then (1) rewrite `siteAddress` (full joined `address, city, state zip`) on every job for that customer inside the same `setData`, and (2) `setContractorJobs`/`saveContractorJobs` to refresh `address/city/state/zip` on every contractor job for that customer (fans out to all clients). Also hardened the auto-mirror in `handleUpdateJob`: seed the new ContractorJob from the full customer record (`mirrorCust` phone/email/city/state/zip), not just `siteAddress` + empty city/zip, and refresh address on the already-mirrored sync branch. PATTERN: contractor WO address has TWO sources - the live `toContractorJobView` (reads `job.siteAddress`) and the stored `ContractorJob` snapshot; a customer-address edit must cascade to BOTH (job.siteAddress + contractorJobs by customerId).
