## Customer address cascade to contractor WO + SO pay-section redesign (commits `ee1a265`, `d2ed3d6`)

### Address edit now reaches the dispatched contractor WO (`ee1a265`)
Bug: editing a customer's address (US-15313) did not update the contractor's Work Order. The contractor view is a LIVE projection `toContractorJobView(job)` which reads `job.siteAddress` (city/state/zip hardcoded ''), and the dispatched `ContractorJob` is a snapshot taken at dispatch; but `handleUpdateCustomer` only rewrote the customer record, so neither updated.
Fix in `App.handleUpdateCustomer`: detect an address change (vs `data.customers` prev), then (1) rewrite `siteAddress` (full joined `address, city, state zip`) on every job for that customer inside the same `setData`, and (2) `setContractorJobs`/`saveContractorJobs` to refresh `address/city/state/zip` on every contractor job for that customer (fans out to all clients). Also hardened the auto-mirror in `handleUpdateJob`: seed the new ContractorJob from the full customer record (`mirrorCust` phone/email/city/state/zip), not just `siteAddress` + empty city/zip, and refresh address on the already-mirrored sync branch. PATTERN: contractor WO address has TWO sources - the live `toContractorJobView` (reads `job.siteAddress`) and the stored `ContractorJob` snapshot; a customer-address edit must cascade to BOTH (job.siteAddress + contractorJobs by customerId).

### Contractor WO address resolves LIVE from the customer (`453a9d2`)
Follow-up to `ee1a265`: even after the cascade, contractor WOs 3/4 still showed "address not located". Root cause: the contractor "My Jobs" view AND the admin Service Orders projection are both built by `toContractorJobView`, which hardcoded `address: job.siteAddress` + `city/state/zip = ''/'FL'/''` and never read the customer. So a WO only had an address if the admin Job's `siteAddress` snapshot happened to be set; a later customer-address edit never surfaced. Fix: `toContractorJobView(job, existingCj, customer?)` now resolves `address/city/state/zip/phone/name` from the linked Customer LIVE, falling back to `job.siteAddress` when unlinked. All 3 call sites (App.tsx 2244 contractor portal, 2292 admin contractorJobsView, 2635 linked-contractor) pass `data.customers.find(c => c.id === j.customerId)`. Now editing the customer reflects on the contractor WO immediately, no WO re-save. PATTERN: the contractor WO is a LIVE projection of the admin Job, not a copy - address/contact should be pulled from the Customer at projection time, not snapshotted.

### Geocode fallback for rural addresses (`3cef23a`)
After `453a9d2` the city resolved (LABELLE/Boyette showed) but WOs 3/4 still rendered "not located". Root cause confirmed against live Nominatim: rural house numbers are absent from OSM - `"6024 ACORN CIR, LABELLE, FL 33935"` -> NO RESULT, while `"LABELLE, FL 33935"` resolves. Located WOs (Venice, Haines City) matched at house/city level. New `geocodeAddress()` in `addressValidator.ts`: (1) tries the full street, de-duplicating any city/state/ZIP already embedded in the street string (avoids `"...33935, LABELLE, FL 33935"` double queries), (2) falls back to the city/ZIP centroid so the stop still plots approximately. GUARD: the fallback requires a city OR state - a bare ZIP `"33935"` geocodes to SPAIN (verified). Both `JobMapView` (dispatch + contractor map) and `SiteMapView` (SO Map tab) now use it; failed lookups were never cached, so the fallback applies on next load with no cache clear. PATTERN: Nominatim returns ZERO results for unknown rural house numbers (not a city fallback) - always retry city/ZIP-with-state; never geocode a bare ZIP.
