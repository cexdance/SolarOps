## DEPLOYED 717e32b - Work Order -> Service Order rename (contractor gets a Work Order)

**Commit:** `717e32b` (pushed to `main`).
**Why:** Domain-language restructure. The end-to-end order is now a SERVICE ORDER. When it is dispatched to a contractor, the contractor receives a WORK ORDER (same number, WO- prefix). The work order closes at the `completed` stage and the service order continues to invoicing + payment.

### Decisions (from the user)
- Rename depth: UI labels + code symbols.
- Numbering: SAME number, SO- prefix for service order, WO- for contractor work order, prefix before the number.
- "Work Order closed" = the existing `completed` stage (no new stage).

### What changed
- **Component**: `WorkOrderPanel.tsx` -> `ServiceOrderPanel.tsx` (git-tracked rename) + export/props `WorkOrderPanel`->`ServiceOrderPanel`, updated 4 importers (App, Jobs, SiteProfilePanel, Customers) via `perl s/WorkOrderPanel/ServiceOrderPanel/g` (safe: that identifier only ever meant the panel).
- **Numbering** (`src/lib/woHelpers.ts`): new `bareOrderNo()`, `serviceOrderNo()` (SO-), `workOrderNo()` (WO-), `generateServiceOrderNumber()` (SO-YYMM-NNNNN). One shared number, two display prefixes. Re-prefixing is DISPLAY-ONLY so legacy `WO-…` records render as `SO-…` in-app with zero data migration.
- **Pipeline**: `WO_STAGES` `completed` label -> "Work Order Closed" / short "WO Closed" (the `'completed'` key/string value is UNCHANGED).
- **Dispatch**: `onDispatch(buildContractorJob(workOrderNo(woNum)))` so the contractor's `ContractorJob.woNumber` is WO-prefixed. The dispatch line-item description stays `Work Order ${woNumber}`.
- **Display wrapping**: every main-app `job.woNumber` render now goes through `serviceOrderNo(...)` (panel header/footer, SOW report, Dashboard, DispatchDashboard, RMADashboard, SiteProfilePanel, Customers, SowDistributionModal, BillingReportModal, LeadLobby + lib billingService/quoteService notifications). QuotePreviewModal label "WO:" -> "SO:". "WO #" headers in BillingReportModal/BillingModule -> "SO #".
- **Label sweep**: ~145 user-facing "Work Order"/"work order" strings across 33 main-app files -> "Service Order" via `grep -rl … | while read f; perl s/Work Order/Service Order/g …`. The contractor portal (`src/components/contractor/*`) was EXCLUDED and keeps "Work Order" (7 refs).

### CRITICAL guardrail (kept intentionally)
- Serialized/synced fields `Job.woStatus`, `woNumber`, `woPhotos` and the `WOStatus` string VALUES (`'quote_sent'`, `'completed'`, …) were NOT renamed - renaming them would break cross-client sync + stored records. Type aliases `WOStatus`/`WOLineItem`/`WOPhoto` and the separate legacy `WorkOrder` mock type in operationsStore also left as-is. So "code symbols" rename = component + local consts + display logic, NOT the data layer.
- Client-status options "WO Assigned"/"WO Completed" in Customers were left as WO (they correctly denote the contractor work-order phase, and the values are persisted).

### Patterns worth reusing
- Spaced "Work Order" is a SAFE bulk-replace target (no code identifier contains a space); the no-space `WorkOrder`/`woStatus`/`WOStatus` identifiers are untouched by it. Restore the 1-2 intentional "Work Order" strings after the sweep.
- Robust multi-file sed: `grep -rl PATTERN … | while IFS= read -r f; do perl -i -pe '…' "$f"; done` (the `for f in $(…)` form mangled here).
- Two-prefix-one-number: store raw, re-prefix at display with helpers; zero migration.

## DEPLOYED 464e145 - Remove Quote Amount + fix comment image-paste crash

**Commit:** `464e145` (pushed to `main`). File: `solarflow-dashboard/src/components/ServiceOrderPanel.tsx`.

### 1. Removed the Quote Amount field
- Deleted the "Quote Amount" `<input>` block (label + $ icon + discount preview + helper text) from the service order overview. Pricing lives in Xero.
- Kept `quoteAmount`/`setQuoteAmount` state (still auto-derived from line items / client-rate buttons and used in handleSave totals + Cost Breakdown). Only the manual UI control was removed. `DollarSign` import still used by the activity-timeline price icon.

### 2. Fixed comment image-paste crash + "image did not save"
- ROOT CAUSE: the comment box reused `handleNotesPaste`, which (a) stored the pasted image as a base64 `dataUrl` in `woPhotos` (synced state) and routed it to the Photos tab, not the comment; (b) on an upload failure the multi-MB base64 stayed in state and reached the save/sync payload, which could bloat storage and hang the panel.
- NEW `handleCommentPaste` (wired to the comment MentionTextarea `onPaste`): upload-FIRST so only a Storage URL ever enters state (never base64); attaches the image to the comment via `Activity.attachments` (`{id,name,url,mimeType}` - ActivityFeed already renders these as thumbnails); fully try/caught with a visible error toast; staged thumbnails with a remove (X) button + a spinner while uploading.
- `addComment` now: allows image-only comments (text OR >=1 attachment), includes `attachments`, clears staged attachments, and persists immediately via `setTimeout(()=>handleSaveRef.current(undefined,true),0)` (comments previously only saved on a later handleSave).
- Hardened `handleNotesPaste` (Notes/Report tabs) too: on upload failure it now `setWoPhotos(prev => prev.filter(p => p.id !== photoId))` so the optimistic base64 never persists.
- Moved the `commentAttachments`/`commentUploading`/`commentError` state declarations ABOVE `addComment` (TDZ: the useCallback dep array referenced them before declaration -> TS2448).

### Patterns
- Image paste into synced React state must be UPLOAD-FIRST (store URL, never base64) - base64 in a synced/persisted payload is the bloat/crash vector. Drop the optimistic copy if the upload fails.
- `Activity.attachments: {id,name,url,mimeType}[]` is the supported way to attach images to a comment; ActivityFeed renders image mimeTypes as thumbnails.

### Verification
- tsc clean, `npm run build` OK, 127/127 tests pass. No live click-through (preview server down); the upload-first + try/catch design makes the handler crash-proof by construction.
